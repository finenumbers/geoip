# Установка

Руководство для оператора: развёртывание GeoIP Analytics через **Docker Compose** (CLI или **Portainer**) с опциональным NGINX Proxy Manager (NPM) для HTTPS.

## Требования

- Docker Engine и Docker Compose v2 (или Portainer CE/BE с поддержкой Compose stacks)
- Доступ к личному кабинету ГРЧЦ GeoIP (email и пароль — у каждого пользователя свои)
- Опционально: домен и NGINX Proxy Manager для HTTPS и Access List

## Способы установки

| Способ | Когда использовать |
|--------|-------------------|
| **[Portainer (Stacks)](PORTAINER.md)** | **Рекомендуется:** UI, GitHub deploy, мониторинг контейнеров |
| [Docker Compose (CLI)](#вариант-1-docker-compose-cli) | SSH на сервер, `docker compose up` |
| [NPM (периметр)](#nginx-proxy-manager) | HTTPS и Access List — см. [NGINX-PROXY-MANAGER.md](NGINX-PROXY-MANAGER.md) |

Оба варианта используют одни и те же файлы: `docker-compose.yml` + `docker-compose.prod.yml`. Файл `.env` **не обязателен** — в compose уже заданы defaults: `POSTGRES_*=geoip`, `CONFIG_MASTER_KEY=""` (автогенерация на volume), образы GHCR `:latest`.

## Вариант 1: Docker Compose (CLI)

### 1. Клонировать репозиторий

```bash
git clone https://github.com/finenumbers/geoip.git
cd geoip
```

### 2. (Опционально) Локальная разработка

```bash
cp .env.example .env
# Только для npm run dev / psql — Docker Compose .env не требует
```

Минимальный bootstrap — только инфраструктура. Секреты ГРЧЦ, API keys и Google Maps задаются в Admin UI.

### 3. Запустить production stack

```bash
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build
```

### 4. Проверить контейнеры

```bash
docker compose ps
curl -s http://localhost:8080/api/v1/health
curl -s http://localhost:8080/api/v1/public/setup-checklist
```

### 5. Первичная настройка

Откройте `http://<хост>:8080/admin/setup`, создайте admin, затем `/admin/setup-api-key`. Дальше — [checklist первого запуска](#первый-запуск-checklist).

### Обновление (CLI)

```bash
cd GeoIP
git pull
docker compose -f docker-compose.yml -f docker-compose.prod.yml up -d --build
```

Volume `config_data` сохраняет настройки между redeploy. Если задан `CONFIG_MASTER_KEY`, secrets останутся расшифровываемыми.

### Остановка и удаление

```bash
# Остановить без удаления данных
docker compose -f docker-compose.yml -f docker-compose.prod.yml down

# Удалить контейнеры и volumes (данные БД и config будут потеряны!)
docker compose -f docker-compose.yml -f docker-compose.prod.yml down -v
```

## Вариант 2: Portainer

> **Полная пошаговая инструкция:** **[PORTAINER.md](PORTAINER.md)** — создание stack из GitHub, переменные окружения, Admin setup, import, обновление и troubleshooting.

Кратко: **Portainer → Stacks → Repository** → compose path **`docker-compose.portainer.yml`**, environment из [`stack.env.example`](../infra/portainer/stack.env.example).

См. также [`infra/portainer/README.md`](../infra/portainer/README.md).

Portainer — удобный способ развернуть тот же Compose stack через веб-интерфейс: не нужен SSH для каждого обновления, видны логи и статус контейнеров.

Подробный гайд также в [`infra/portainer/README.md`](../infra/portainer/README.md).

### Предварительные условия

1. Portainer установлен и имеет доступ к Docker на целевом хосте.
2. Репозиторий GeoIP доступен на хосте (git clone) **или** вы вставляете compose в Web editor (см. ниже).

Portainer подключает overlay через [`infra/portainer/stack.compose.yml`](../infra/portainer/stack.compose.yml), который включает корневые `docker-compose.yml` и `docker-compose.prod.yml`.

### Способ A: Stack из Git-репозитория (рекомендуется)

1. **Portainer → Stacks → Add stack**
2. **Name:** `geoip` (или своё имя)
3. **Build method:** Repository
4. **Repository URL:** URL вашего GitHub/GitLab репозитория
5. **Repository reference:** `main` (или нужная ветка)
6. **Compose path:** `infra/portainer/stack.compose.yml`
7. **Environment variables** — опционально, см. [`stack.env.example`](../infra/portainer/stack.env.example) (defaults уже в compose)

8. **Deploy the stack**

Portainer поднимет сервисы: `postgres`, `pgbouncer`, `api`, `import`, `export`, `web`.

### Способ B: Web editor (ручная вставка)

Если репозиторий уже склонирован на сервер (например, в `/opt/GeoIP`):

1. **Portainer → Stacks → Add stack → Web editor**
2. Вставьте содержимое [`infra/portainer/stack.compose.yml`](../infra/portainer/stack.compose.yml)
3. Укажите **Working directory** / путь к репозиторию на хосте, чтобы `include` нашёл `../../docker-compose.yml`
4. Задайте environment variables (как в способе A)
5. **Deploy the stack**

> **Важно:** `stack.compose.yml` использует `include` с относительными путями. Рабочая директория stack в Portainer должна соответствовать расположению файла `infra/portainer/stack.compose.yml` внутри клонированного репозитория.

### После деплоя в Portainer

1. **Stacks → geoip →** убедитесь, что все контейнеры `running` (первый старт API может занять 1–2 минуты из‑за migrations).
2. Откройте `http://<docker-host>:8080/admin/setup`.
3. Выполните [checklist первого запуска](#первый-запуск-checklist).

### Смена пароля Postgres в Portainer

1. Обновите `POSTGRES_PASSWORD` в environment variables stack.
2. Синхронизируйте [`infra/pgbouncer/userlist.txt`](../infra/pgbouncer/userlist.txt) (формат `"user" "md5..."`).
3. **Update the stack** (redeploy).

### Обновление stack в Portainer

- **Git method:** Pull and redeploy (или webhook при push в main).
- **Web editor:** Update stack → redeploy после `git pull` на хосте.

Volume `config_data` не удаляется при redeploy — настройки Admin сохраняются.

### Логи и диагностика в Portainer

- **Containers → geoip_api / geoip_import / geoip_web → Logs**
- Или на хосте: `docker compose -f docker-compose.yml -f docker-compose.prod.yml logs -f import`

Проверки:

```bash
curl -s http://<host>:8080/api/v1/public/setup-checklist
curl -s http://<host>:8080/api/v1/ready
```

## Первый запуск (checklist)

После любого способа установки база данных **пустая** — это ожидаемо. На Dashboard и в Admin → Обзор отображается checklist:

1. **Admin** — `/admin/setup`: логин и пароль локального администратора
2. **API-ключ External IP Lookup** — `/admin/setup-api-key`: Generate → Save (обязательно)
3. **ГРЧЦ** — Admin → ГРЧЦ / Import: email и пароль ЛК, сохранить, «Проверить ГРЧЦ»
4. **Импортировать датасет** — Admin → Обзор → «Импортировать датасет» (или дождаться cron **10:00 MSK**)
5. **Google Maps** (опционально) — Admin → Интеграции

Пока import не завершён успешно, `/api/v1/ready` возвращает `not_ready` — это нормально.

**Не используйте** `seed:fixture` в production — только для локальной разработки и CI.

## Состав stack (Docker Compose)

| Сервис | Назначение |
|--------|------------|
| `postgres` | База данных |
| `pgbouncer` | Connection pooling |
| `api` | HTTP API + migrations on startup |
| `import` | Import worker + cron |
| `export` | Export worker |
| `web` | Nginx + SPA (порт **8080** на хосте в prod) |

## Volumes

| Volume | Назначение |
|--------|------------|
| `pg_data` | PostgreSQL |
| `config_data` | `settings.json`, `secrets.enc`, `proxy.env` |
| `import_data` | Временные файлы import worker |
| `export_data` | CSV-экспорты |

## NGINX Proxy Manager

Подробная инструкция: **[NGINX-PROXY-MANAGER.md](NGINX-PROXY-MANAGER.md)**

Кратко (CLI и Portainer):

1. **Proxy Host:** домен → `http://<docker-host>:8080`, схема **HTTP**
2. **SSL:** Let's Encrypt на NPM (не на контейнере)
3. **Access List:** ограничьте доступ к UI и `/admin`
4. Не публикуйте порты API (`3000`) и Postgres на хост — prod overlay оставляет только `:8080`

После настройки NPM открывайте `https://<ваш-домен>/admin/setup`.

## Переопределение defaults при деплое

| Параметр | Где | Зачем |
|----------|-----|-------|
| `POSTGRES_*` | правка `docker-compose*.yml` (+ pgbouncer userlist для dev) | Сильный пароль БД |
| `CONFIG_MASTER_KEY` | compose env или Portainer | 64 hex — стабильное шифрование `secrets.enc` |
| Image tag | замена `:latest` в compose | Pin версии GHCR |

Операционные секреты (ГРЧЦ, API keys, Google Maps) — только через **Admin UI**.

## Восстановление данных

Автобэкап PostgreSQL не используется. При потере или повреждении данных выполните **повторный import** датасета из ГРЧЦ (Admin → Обзор).
